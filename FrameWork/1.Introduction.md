# 前端框架入门讲义

## 一、我们每天看到的网站是怎么来的？

一个最简单的网站，本质上由三部分组成：

- **HTML**：页面结构（有什么）
- **CSS**：页面样式（长什么样）
- **JavaScript**：页面逻辑（能做什么）

这三样，被称为 **前端三大件**。

在早期，开发网页基本就是：

- 写一堆 HTML
- 用 CSS 调样式
- 用 JavaScript 操作 DOM，让页面“动起来”

这种方式，被称为：
👉 **传统前端开发方式（原生 HTML + CSS + JS）**

---

## 二、传统写法真的不好吗？

### 先说结论

**不是不能用，而是“不适合复杂项目”**

### 传统写法的优点

- 上手快
- 不需要额外工具
- 小页面、简单展示非常合适

### 但当项目变复杂时，问题会迅速出现

#### 1️⃣ 页面一复杂，代码就失控

- HTML 写一大堆
- JS 到处 `getElementById`
- 逻辑和结构混在一起

最终效果：

- 看不懂
- 不敢改
- 一改就崩

---

#### 2️⃣ 状态变化很难维护

例如一个常见需求：

> 登录状态改变 →
> 导航栏、头像、按钮、权限全部要跟着变

在传统 JS 里：

- 你要手动找到每个地方
- 一个一个改 DOM
- 很容易漏掉

页面状态 ≠ 数据状态
**不同步问题极其常见**

---

#### 3️⃣ 团队协作几乎是灾难

- 同一个 HTML 文件几千行
- 不知道谁改了哪里
- 冲突频繁
- 维护成本极高

---

## 三、前端框架是为了解决什么问题？

### 一句话理解

👉 **前端框架 = 用“更合理的方式”来组织网页和逻辑**

它们解决的核心问题是：

- 页面越来越复杂
- 交互越来越多
- 代码需要长期维护
- 需要多人协作

---

## 四、什么是「前端框架」？

### 不严谨但好理解的定义

> 前端框架是一套 **帮你写网页的“规则 + 工具”** ，有规则的不自由换来的是更加自由

它会帮你处理：

- 页面结构怎么拆
- 数据如何驱动界面
- 状态变化怎么更新页面
- 项目如何组织

---

## 五、前端框架的核心思想（非常重要）

### 1️⃣ 5.1. 组件化（Component）

#### 传统方式

- 一个页面写完
- 复制粘贴
- 改一点就出 bug

#### 框架方式

- 页面拆成一个个「组件」

  - 按钮
  - 卡片
  - 表单
  - 弹窗

组件的特点：

- 可复用
- 可组合
- 各管各的

👉 **像搭积木一样做网页**

---

### 2️⃣ 5.2. 数据驱动视图

这是现代前端最重要的思想之一。

#### 传统 JS

- 你改数据
- 再手动改 DOM

#### 框架

- 你只管改数据
- 页面自动更新

**数据 = 页面状态变更的唯一来源**

这让：

- 逻辑更清晰
- Bug 更少
- 页面行为更可预测

---

### 3️⃣ 5.3. 关注点分离（但不是文件分离）

以前：

- HTML 一个文件
- CSS 一个文件
- JS 一个文件

但逻辑是混在一起的。

框架强调：

- **一个功能 = 一个组件**
- 结构、样式、逻辑围绕“功能”组织

👉 不是“按技术分”，而是“按功能分”

---

## 六、为什么现在企业几乎都用框架？

### 从企业角度看

- 项目生命周期长
- 需要不断迭代
- 多人协作是常态
- 新人要能快速接手

### 框架带来的实际价值

- 代码更可维护
- 更容易测试
- 更容易扩展
- 更容易招人（生态统一）

---

## 七、那是不是再也不用原生 JS 了？

### 不是 ❌

而是：

- **框架是建立在原生 JS 之上的**
- 不懂 JS，用框架会非常痛苦
- 框架 ≠ 魔法

你可以理解为：

> 原生 JS 是内功
> 前端框架是招式

---

## 八、常见前端框架（后面会讲，现在过一下）

---

### React (目前全世界使用量第一)

- 思想影响力最大
- 组件化非常彻底
- 生态极其丰富
- 学会后可做 Web / App / 桌面端

> 历史: React 起源于 Facebook 的内部项目 (当时这个还是一个 PHP 语言的一个插件 XHP)，因为该公司对市场上所有 JavaScript MVC 框架，都不满意，就决定自己写一套，用来架设 Instagram 的网站。做出来以后，发现这套东西很好用，就在 2013 年 5 月开源了。

👉 企业使用率非常高

---

### Vue (目前全世界使用量第二)

- 上手友好
- 语法直观
- 学习曲线平缓
- 国内使用非常广泛 (创始人: 尤雨溪，Google 就职，动漫爱好者，现在在新加坡)
- (补充: 构建工具 Vite 也是他开发的)

> 来自维基百科: 尤雨溪之后将所开发出的框架命名为“Vue.js”，并在 2014 年推出第 1.0 版，获得回响之后成为代码托管服务平台 GitHub 上最多人关注的项目之一。另出自于对二次元动漫的喜爱，尤雨溪在推出的 Vue.js 各版本里，会以动漫作品当作代号。如第 1.0 版的“Evangelion”，是《新世纪福音战士》；第 2.0 版的“Ghost in the Shell”，是指《攻壳机动队》；第 2.3 版的“JoJo's Bizarre Adventure”，则是《JoJo 的奇妙冒险》;第 3.2 版的“Quintessential Quintuplets”，则是《五等份的花嫁》等等。

👉 非常适合初学者

---

### Angular

- 框架本身非常完整
- 学习成本高
- 更偏工程化

👉 多见于大型企业项目 (现在也基本不用转型为 React 了)

---

### 现代全栈框架（后面不仅要学，还要使用）

- Next.js
- Nuxt.js

特点：

- 在前端框架基础上
- 加入工程能力（构建、路由、渲染方式等）

---

## 九、学习路线的正确心态

- 不要一上来就觉得“框架很难”
- 它本质是在**帮你减少混乱**
- 学的是“思想”，不是 API 工具

当你真正理解了：

- 组件
- 状态
- 数据驱动

你会发现：

> **不用框架反而更难**
